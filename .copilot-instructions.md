# FlutterBase — Copilot Instructions

You are assisting in a Flutter project called **FlutterBase**.
This is a gamified learning platform (Duolingo-style) focused on teaching Flutter through interactive quizzes, XP, streaks, levels, daily challenges, and a progress dashboard.

Generate **high-quality, scalable, and maintainable** code that follows the conventions established in this codebase.

---

## Project Overview

| Aspect | Detail |
|---|---|
| **State Management** | Bloc (`flutter_bloc`) |
| **DI** | Riverpod providers (no service locators) |
| **Routing** | `go_router` (declarative, path-based) |
| **Data Storage** | Local-only (in-memory / `shared_preferences`). No remote API. |
| **Code Generation** | `freezed` + `json_serializable` (models), `slang` (i18n) |
| **Linting** | `very_good_analysis` (strict) |
| **Fonts** | `google_fonts` (JetBrains Mono + Inter) |
| **i18n** | `slang` / `slang_flutter` — JSON-based, type-safe |
| **Theme** | Dark-only Material 3, custom `AppColors` / `AppTextStyles` / `AppTheme` |

## Architecture

### Pattern — Clean Architecture (Feature-First)

```
lib/
├── app/                          # App-level setup
│   ├── assets/                   # i18n JSON, images
│   ├── providers/                # Top-level Riverpod providers
│   ├── routes/                   # GoRouter config + AppRoutes
│   └── theme/                    # AppColors, AppTextStyles, AppTheme
├── core/                         # Shared across all features
│   ├── constants/
│   ├── errors/                   # Exception & Failure classes
│   └── utils/                    # Extensions, helpers
├── features/
│   └── <feature_name>/
│       ├── data/
│       │   ├── datasources/      # Local data sources only
│       │   ├── models/           # Freezed models + toEntity/toModel mappers
│       │   └── repositories/     # Repository implementations
│       ├── domain/
│       │   ├── entities/         # Plain Dart classes (no dependencies)
│       │   ├── repositories/     # Abstract repository contracts
│       │   └── usecases/         # Single-purpose business logic
│       └── presentation/
│           ├── providers/        # Feature-scoped Riverpod providers/notifiers
│           ├── pages/            # Full-screen widgets (ConsumerWidget)
│           └── widgets/          # Reusable UI components
└── gen/                          # Auto-generated (i18n strings)
```

### Data Flow — Always respect these layers

```
UI (Page/Widget) -> Provider/Notifier -> UseCase -> Repository (abstract) -> DataSource
```

- **Never** skip layers (e.g., never call a DataSource from a Notifier).
- **Domain layer has zero dependencies** on data or presentation.
- Notifiers depend on **UseCases**, not Repositories directly.

## Design Principles

### SOLID

- **S** — One class, one reason to change. Each UseCase does one thing.
- **O** — Extend behaviour through new classes, not modifying existing ones.
- **L** — Subtypes must be substitutable (e.g., `ProgressRepositoryImpl` for `ProgressRepository`).
- **I** — Keep interfaces small and focused.
- **D** — Depend on abstractions (`ProgressRepository`), never on concretions (`ProgressRepositoryImpl`).

### General

- **DRY** — Extract shared logic into `core/utils` or shared widgets.
- **KISS** — Favour simplicity. Don't add abstractions unless they reduce complexity.
- **Composition over inheritance** — Prefer mixing small widgets/classes.
- **No premature optimization** — Optimize only when measurable.

## State Management — Riverpod

- Use **`Notifier`** for synchronous state and **`AsyncNotifier`** for async state.
- State objects must be **immutable** (use `copyWith` or Freezed).
- Pages extend **`ConsumerWidget`** or **`ConsumerStatefulWidget`** — access state via `ref.watch()` / `ref.read()`.
- Keep notifiers **thin**: business logic belongs in **UseCases**.
- Avoid `setState` unless trivial, UI-only animation state.

```dart
// Provider definition
final progressProvider =
    AsyncNotifierProvider<ProgressNotifier, UserProgress>(
  ProgressNotifier.new,
);

// Notifier example
class ProgressNotifier extends AsyncNotifier<UserProgress> {
  late final GetUserProgress _getUserProgress;
  late final UpdateXp _updateXp;

  @override
  Future<UserProgress> build() async {
    _getUserProgress = ref.read(getUserProgressProvider);
    _updateXp = ref.read(updateXpProvider);
    return _getUserProgress();
  }

  Future<void> addXp(int amount) async {
    await _updateXp(amount);
    state = AsyncData(await _getUserProgress());
  }
}
```

```dart
// In a page
class ProgressPage extends ConsumerWidget {
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final asyncProgress = ref.watch(progressProvider);
    return asyncProgress.when(
      data: (progress) => _buildContent(progress),
      loading: () => const CircularProgressIndicator(),
      error: (e, st) => Text('Error: $e'),
    );
  }
}
```

## Dependency Injection — Riverpod Providers

- Every dependency (datasource, repository, usecase) is a **Riverpod `Provider`**.
- Register providers in `lib/app/providers/` or co-located in the feature's `presentation/providers/`.
- **Never** instantiate dependencies manually inside classes.

```dart
// Datasource
final progressLocalDatasourceProvider = Provider<ProgressLocalDatasource>(
  (ref) => ProgressLocalDatasource(),
);

// Repository (bind abstraction to implementation)
final progressRepositoryProvider = Provider<ProgressRepository>(
  (ref) => ProgressRepositoryImpl(ref.read(progressLocalDatasourceProvider)),
);

// UseCase
final getUserProgressProvider = Provider<GetUserProgress>(
  (ref) => GetUserProgress(ref.read(progressRepositoryProvider)),
);
```

## Routing — GoRouter

- Declare all route paths as **static constants** in `AppRoutes`.
- Use **`context.go()`** / **`context.push()`** for navigation.
- Dynamic routes use path parameters (e.g., `/quiz/:level`).

```dart
context.go(AppRoutes.quizByLevel(2));
```

## Data Storage — Local Only

- This project uses **local-only storage** (no remote API, no backend).
- DataSources: in-memory caches or `shared_preferences` for persistence.
- There is **no network layer** — no `ApiClient`, no `Dio`, no HTTP calls.

---

## Models & Entities

### Domain Entity (plain Dart)
- No annotations, no dependencies. Immutable with `const` constructor and manual `copyWith`.

### Data Model (Freezed)
- Annotated with `@freezed`. Lives in `data/models/`.
- Provides `fromJson` / `toJson` via `json_serializable`.
- **Always** provide mapper extensions: `toEntity()` and `toModel()`.

After changing a model, run code generation:
```bash
dart run build_runner build --delete-conflicting-outputs
```

---

## Error Handling

- Use **`Exception`** subclasses (`CacheException`) in the data layer.
- Map them to **`Failure`** subclasses (`CacheFailure`) at the repository boundary.
- Notifiers expose error states via Riverpod's `AsyncValue.error`.

---

## Internationalisation (i18n) — Slang

- Source files: `lib/app/assets/i18n/{locale}.i18n.json` (currently `en` and `pt`).
- Access translations via **`context.t`** (e.g., `context.t.common.pageNotFound`).
- After editing JSON files, regenerate with `dart run slang`.

---

## Theming

- **Dark-only** theme using Material 3.
- Colours: `AppColors` — never use raw `Color(0x...)` in widgets.
- Text styles: `AppTextStyles` — never inline `TextStyle` in widgets.
- Theme: `AppTheme.dark` — applied in `MaterialApp.router`.
- Typography: **JetBrains Mono** (headings/code) + **Inter** (body) via `google_fonts`.

## Coding Conventions

| Rule | Detail |
|---|---|
| **Naming** | `snake_case` files, `PascalCase` classes, `camelCase` members |
| **Imports** | Use package imports (`package:flutter_base/...`), not relative |
| **Abstract classes** | Use for contracts (repositories, datasources) |
| **`const`** | Apply `const` constructors and `const` values wherever possible |
| **Private fields** | Prefix with `_`; expose only what's needed |
| **File length** | Keep files under 200 lines; split if larger |
| **Widget decomposition** | Extract widgets into `widgets/` when they exceed ~50 lines or are reused |
| **Comments** | Only where the *why* isn't obvious; avoid restating the code |
| **Lint rules** | Follow `very_good_analysis`; never add `// ignore` without justification |

## When Adding a New Feature

1. Create the folder structure under `lib/features/<feature_name>/` with `data/`, `domain/`, and `presentation/` layers.
2. Define the **entity** in `domain/entities/`.
3. Create the **repository contract** in `domain/repositories/`.
4. Implement **use cases** in `domain/usecases/` (one class per use case).
5. Create the **model** (Freezed) in `data/models/` with `toEntity()` / `toModel()` mappers.
6. Implement the **data source** (local only) in `data/datasources/`.
7. Implement the **repository** in `data/repositories/`.
8. Create **Riverpod providers** in `presentation/providers/` (datasource -> repository -> usecase -> notifier).
9. Add **routes** in `app_router.dart` and constants in `AppRoutes`.
10. Build **pages** (`ConsumerWidget`) and **widgets** in `presentation/`.
11. Add i18n keys to each `*.i18n.json` and run `dart run slang`.
